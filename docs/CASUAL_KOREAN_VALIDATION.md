# 자연스러운 한국어 대화 릴리스 게이트

## 목적

형식적인 문장 수·종료 토큰 검사는 답변이 실제 대화에서 자연스럽다는 것을 보장하지
않는다. v0.3.1에서는 짧은 협업 제안과 후속 질문이 모델 후보 폐기 후 고정된
`quality_fallback` 문구로 끝났지만, 기존 검증 corpus에는 이 사용 흐름이 없었다.

`scripts/validate_agent_casual_korean.py`는 이 누락을 재현하는 소스 수준 진단
harness다. 규칙 기반 관측만 사용하며 또 다른 언어 모델로 답변을 채점하지 않는다.
현재 릴리스에서 실제 실행 가능한 제품 게이트는 같은 범주를 포함한 통합 20턴
`product-e4b-it-20` 하나뿐이다.

## 정확한 결함 재현

동일한 세션에서 다음 두 문장을 순서대로 실행한다.

1. `그럼, 나와함께 재미있는 프로젝트를 합시다.`
2. `나와 어떤 일을 함께 할 수 있나요?`

두 문장 모두 제품이 모델·Fact-book보다 먼저 적용하는 제한된
`generation_mode=conversation_fastpath`를 타야 한다. 이 경로는 인사, 협업 제안,
능력 질문, 첫 단계 질문만 처리하고 일반 지식·코드·형식 요청은 정상 Cogni-Core로
넘긴다. 고정 안전 문구, `quality_fallback`, 두 개 이상의 assistant 메시지, 미완성
문장 또는 제한시간 초과는 실패다.

## 검증 범위

총 10턴이 다음 범주를 모두 포함한다.

- 보고된 2턴의 정확한 재현과 동일 세션 후속 질문
- 독립 세션의 일상 인사
- 협업 제안과 능력 질문의 바꿔 말하기
- `재미잇는`, `프로잭트`, `만드러볼래요` 같은 실제 오타
- 대화 문맥을 이어받는 다음 단계 질문
- 프로젝트 대화에서 Python 설명으로 전환하는 문맥 전환
- 세 문장 형식 요청과 Runtime Fact-book 회귀

## 실행

직접 실행은 `BLOCKED`다. E4B-it 전용 guard profile은 구현되어 있지만 이 10턴
validator argv는 의도적으로 guarded `run` allowlist에 넣지 않았다. 더 작은 진단
corpus가 제품 릴리스 증거로 오인되는 것을 막기 위해서다. `docker-argv`는 명령 모양만
검사하며 실행이나 증거 생성을 하지 않는다. 현재 scope의 제품 대화 증거는
`docs/VALIDATION.md`에 적힌 통합 exact-20 명령으로만 생성한다.

증거 JSON은 소스 트리 밖에 원자적으로 기록한다. 제품 대화용 E4B-it 모델과
manifest의 일곱 파일은 로드 전에 검증되며, 네트워크를 사용하지 않는다. pretrained
base 체크포인트는 이 제품 대화 게이트의 입력으로 허용하지 않는다.

## 합격 조건

다음 조건을 모두 만족해야 `strict_casual_gate_passed=true`가 된다.

1. 10턴 전부 완료되고 각 턴의 모든 boolean 검사가 참이다.
2. `quality_fallback_turns=0`이다.
3. 사용자 턴마다 새 assistant 메시지가 정확히 하나다.
4. 역할·제어 토큰, 동일 문장 반복, 이전 턴과 완전히 같은 답변이 없다.
5. 한국어 문장이 자연스러운 문장부호에서 끝나고 요청 형식을 충족한다.
6. 상투적인 복구 문구를 실제 답변으로 오인하지 않는다.
7. 응답은 8~2,000자이며 단어 다양성 0.20 이상이다.
8. 모든 턴이 CLI로 지정한 제한시간 안에 끝난다. 기본 상한은 120초다.
9. 각 사례의 최소 관련 키워드를 포함하고 기대한 Fast Path/Fact-book/Cogni-Core
   경로를 탄다.
10. 종료 시 모델 워커와 GPU lease가 모두 정리된다.

단어 다양성과 키워드는 퇴행을 잡기 위한 결정론적 하한이지, 주관적인 답변 품질
점수나 사실성 인증이 아니다. 이 합격 조건은 진단 코드 자체의 계약이다. 릴리스
승인은 이 harness를 별도로 실행해서 얻지 않으며, 같은 회귀 범주를 포함한 통합
exact-20 결과와 전체 회귀 테스트, 실제 UI HTTP 경로 검증을 함께 요구한다.

## UI 표시 원칙

`quality_fallback`은 정상 답변이 아니다. UI는 이를 `품질 검증 실패 · 복구 필요`로
표시하고 상단 상태도 `RESPONSE FAILED`로 드러낸다. Fact-book 응답은
`FACT-BOOK · 검증된 사실`, 제한된 소셜 응답은 `대화 FAST PATH · 완료`, 실제 생성은
일반 완료로 서로 구분한다.
